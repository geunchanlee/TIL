# TIL 26/02/25 ~ 26 - 데이터 모델링

## 데이터 모델

> 데이터 모델
> 데이터베이스 구조를 단순화, 추상화해 체계적으로 표현하는데 사용되는 도구

### 데이터 모델 구성요소

1. 데이터 구조(Data Structure)
	- 데이터베이스에 저장할 데이터에 대해 요소, 관계를 구조적으로 명세해 표현한 것
2. 연산(Operation)
	- 데이터 구조에 따라 실제 값들을 처리하는 작업에 대한 명세, 데이터 조작 기법
3. 제약조건(Constraint)
	- 데이터 조작 한계를 규정한 것
	- 연산 적용이 허용되는 제약사항

## 개념적 데이터 모델

- 개념적 데이터 모델
	- 데이터베이스의 논리적 구조 명시.
	- 추상화 수준이 높다.
	- ER모델(Entity-Relationship model)이 대표적
- 개체-관계 모델
	- 개체와 개체간의 관계를 이용해 개념적 구조료 표현하는 방법
	-  개념적으로 모델링해 그림으로 표현한 것을 개체-관계 다이어그램(**ERD**, *Entity-Relationship Diagram*)이라고 한다.
	- `ERD` 잘 기억하기

### 개체(entity)

식별자가 있고, 2개 이상 인스턴스가 있으며, 속성을 갖고, 다른 개체와 최소 한 개 이상 관계가 있다. ERD에서 네모로 표현한다.
1. 종류
	- 유형 엔티티: 유형의 정보를 갖는, 업무에서 도출되는 물리적으로 존재하고 지속적으로 사용되는 엔티티
		- 서점의 책
	- 개념 엔티티: 물리적인 형태 존재하지 않고, 개념적으로만 존재하고 사용되는 엔티티
		- 온라인 쇼핑의 장바구니
	- 사건 엔티티: 비즈니스 프로세스 실행 시 생성되는 엔티티
		- 결제내역, 구매내역("결제" 라는 비즈니스 프로세스 실행 시 생성됨)
2. 발생 시점에 따른 종류
	- 기본 엔티티: 업무에 존재하는 정보; 다른 엔티티와 관계에 이존하지 않고 독립적으로 생성됨.
	- 중심 엔티티: 업무 중심역할. 기본 엔티티에서 발생. 데이터양 많음. 다른 엔티티와 관계를 통해 사건엔티티 생성
	- 사건 엔티티: 두 개 이상의 엔티티로부터 발생됨. 내용이 변경되거나 데이터 양이 증가됨(지속적으로 쌓임)

### 속성(attribute)

엔티티가 가지고 있는 특징이나 성질. ERD에서 원으로 표현한다.
속성값은 개체 특성이나 상태가 현실화 된 값. -> 하나의 값만 가진다.
> 책 = 개체(entity)
> ISBN, 제목, 저자, 가격, 목차, 출판사 = 속성

1. 분해 여부에 따른 종류
	- 단일 속성: 하나의 의미로 구성된 것
		- 회원ID, 이름 등
	- 복합 속성: 독립적 의미를 가진 몇 개의 기본적인 단순 속성으로 분해할 수 있는 것
		- 주소(우편번호, 기본주소, 상세주소), 생년월일(연-월-일) 등
	- 다중값 속성: 하나의 속성이 여러 개의 속성값을 가지는 것
		- 연락처(이메일, 전화번호, sns) 등
2. 특성에 따른 종류
	- 기본 속성: 비즈니스 프로세스에서 도출되는 본래의 속성
		- 회원ID, 이름 
	- 설계 속성: 데이터 모델링 과정에서 발생되는 속성
		- 상품코드
	- 파생 속성: 다른 속성에 의해 만들어지는 속성
		- 합계, 평균 

### 관계(relationship)

엔티티와 엔티티가 맺고 있는 의미 있는 연관성.
존재의 행태, 행위로 서로에게 영향준다. ERD에선 마름모로 표현한다.

- 종류
	- 존재 관계: 엔티티 간 상태를 의미
	- 행위 관계: 엔티티 간 어떤 행위가 있는 것
- 관계 차수
	- 특정 관계에 연결된 개체의 개수를 의미한다.
- **1:1 관계**
	- 관계 참여하는 각각의 인스턴스가 관계를 맺는 다른 인스턴스에 대해 하나의 관계만 가지는 것
- **1:N 관계**
	- 관계에 참여하는 각각의 인스턴스가 관계를 맺는 다른 개체의 인스턴스에 대해 하나 이상의 관계를 갖는다. 반대 방향은 하나의 관계만 갖는다.
- **N:M 관계**

## 관계 데이터 모델 

개념적 구조를 논리적 구조로 표현하는 논리적 데이터 모델

| 고객아이디 | 고객이름 | 나이 |   등급   |   직업   | 적립금 |
|:----------:|:--------:|:----:|:--------:|:--------:|:------:|
|  CHAR(20)  | CHAR(20) | INT  | CHAR(10) | CHAR(10) |  INT   |
| thief  |  홍길동  |  40  |   vip    |  회사원  |  1000  |
|    chan    |  이근찬  |  28  |  silver  |   NULL   |  100   | 
|  monster   |  류현진  |  38  |   gold   | 야구선수 |   99   |

1번 줄: 열(속성, 애트리뷰트) | 2번 줄: 도메인 | 4, 5번 줄 행(튜플)
- **릴레이션(Relation)**: 하나의 개체에 관한 데이터를 2차원 테이블 구조로 저장한 것
- **속성(Attribute)**: 릴레이션의 열(column)
	- 6개의 세로열
- **튜플(Tuple)**: 릴레이션의 행(row)
	- 홍길동 줄, 이근찬 줄 등 3개
- **도메인(Domain)**: 하나의 속성이 가질 수 있는 값들의 집합
	- 속성 "등급"의 도메인: `{vip, gold, silver}`
- **NULL**: 아직 모르거나 해당 사항이 없음을 표현하는 특별한 값
	- `0`, `" "`과 같은 게 아니라 값이 없음을 표현한다.
- **차수(Degree)**: 하나의 릴레이션에서 **속성** 전체 개수 
	- 이 릴레이션에선 `6` (고객아이디 ~ 적립금)
- **카디널리티(Cardinality)**: 하나의 릴레이션에서 **튜플**의 전체 개수
	- 이 릴레이션에선 `3`

### 릴레이션의 특성

- **튜플의 유일성**
	- 하나의 릴레이션엔 동일한 튜플은 존재할 수 없다.
	- 튜플을 유일하게 구별하기 위한 속성을 키(key)라고 한다.
	- 키를 이용해 튜플의 유일성이 만족되면 릴레이션에서 튜플에 쉽게 접근할 수 있다.
- **튜플의 무순서**
	- 하나의 릴레이션에서 튜플의 순서는 무의미하다.
- **속성의 무순서**
	- 하나의 릴레이션에서 속성 사이의 순서는 무의미하다.
	- 릴레이션에서 속성값의 위치가 아닌 이름으로 접근하기 때문에 무의미
- **속성의 원자성**
	- 속성 값으로 원자 값만 사용 가능하다.
	- 모든 속성 값은 더 분해할 수 없는 하나의 값만 가질 수 있다.(다중 값 x)


### 키의 종류

`key` = 데이터를 식별하는 고유한 식별자(identifier)기능을 하는 것

1. **슈퍼키(Super Key)**
	- DB에서 관계의 행을 고유하게 식별할 수 있는 속성 또는 속성의 집합()
	- 유일성(uniqueness) = 하나의 키 값으로 하나의 행을 유일하게 식별할 수 있는 성질
	- 슈퍼키는 유일성을 만족한다.
2. **후보키(Candidate Key)**
	- 속성들 중 각 행을 유일하게 식별할 수 있는 최소한의 속성들의 집합
	- 최소성(minimality) = 모든 행을 유일하게 식별할 수 있는 최소한의 속성들의 집합
	- 후보키는 유일성, 최소성을 만족한다.
3. **기본키(Primary Key, PK)**
	- 여러 후보키 중에 하나를 선택해 행을 식별하기 위한 기준으로 사용하는 키
	- 기본키로 정의된 속성은 NULL이나 중복값 가질 수 없다. (개체 무결성 제약조건)
4. **대체키(Alternate Key)**
	- 후보키가 두 개 이상인 경우에 기본키로 하나를 지정하고 남은 후보키
5. **외래키(Foreign Key, FK)**
	- 어떤 릴레이션의 기본키를 참조하는 속성
	- 참조 무결성 확인을 위해 사용되는 키
	- 허용된 데이터값만 데이터베이스 저장을 위해 사용


### 관계 데이터 모델의 제약: 무결성 제약조건

데이터베이스 상태가 만족해야할 조건
권한 가진 사용자로부터 DB의 정확성을 지키는 수단

- **개체 무결성 제약조건**
	- 기본키(Primary Key) 제약이라고도 한다.
	1. 릴레이션에서 기본 키의 속성은 NULL이나 중복 값을 가질 수 없다.
- **참조 무결성 제약조건**
	- 외래키(Foreign Key) 제약이라고도 한다.
	- 릴레이션 간의 참조관계를 선언하는 제약조건
	1. 자식 릴레이션의 외래키는 부모 릴레이션의 기본키와 도메인이 동일해야 한다.
	2. 자식 릴레이션 값이 변경될 때 부모 릴레이션의 제약을 받는다.
		- 부모 릴레이션 도메인과 다른 값으로 삽입, 수정할 때 거부됨
		- 자식 릴레이션에서 참조하는 값을 부모에서 삭제하거나 변경하려고 할 때 거부됨

## 데이터베이스 설계

### 데이터베이스 설계 과정

> **1단계**: 요구사항 분석
> - 데이터베이스 용도 파악, 데이터 간 관계 이해, 비즈니스 규칙 정의
> - 결과물: `요구사항 명세서`

> **2단계**: 개념적 설계
> - 독립적인 개념적 구조 설계, 엔티티와 관계 식별
> - 결과물: `개념적 스키마(ER 다이어그램 작성)`

> **3단계**: 논리적 설계
> - DBMS에 적합한 논리적 구조 설계, 개념적 모델을 관계형 모델로 변환
> - 결과물: `논리적 스키마(테이블 구조 정의)`

> **4단계**: 물리적 설계
> - DBMS로 구현 가능한 물리적 구조 설계
> - 결과물: `물리적 스키마(인덱스, 저장 방식 등 결정)`

> **5단계**: 구현
> - SQL문 작성 후 이를 DBMS에서 실행해 데이터베이스 생성


### ERD(Entity Relation Diagram)

데이터베이스 객체 중 테이블과 뷰 간의 관련성을 표시한 그림

데이터의 흐름을 정리한 DFD(Data Flow Diagram)도 있다.

> [!Note] 기본 구성 요소

- 엔티티(Entity)
	- :LiSquare: `사각형`으로 표현
	- 데이터베이스에서 테이블로 변환된다
- 속성
	- 엔티티의 특성이나 성질
	- 테이블의 컬럼이 된다
- 관계(Relationship)
	- 엔티티 간의 연결을 `선`으로 표시
	- :LiSquare::LiLineHorizontal::LiSquare:

> [!Note] 관계의 종류

- 일대일(1:1) 관계
	- 한 엔티티의 하나의 레코드가 다른 엔티티의 레코드 하나에 연결
	- ex) 사용자와 사용자 프로필
- 일대다(1:N) 관계
	- 한 엔티티 하나의 레코드가 다른 엔티티의 여러 레코드와 연결
	- 가장 흔한 관계 유형이다
	- ex) 부서와 직원(한 부서에 직원 여럿이 소속됨)
- 다대다(N:M) 관계
	- 양쪽 엔티티의 여러 레코드가 서로 연결
	- ex) 학생과 강의(학생은 여러 강의 수강, 강의는 여러 학생이 수강)

**ERD 관계 설정 시 고려사항**
- 기본 키(Primary Key) 선택
	- 자연키(Natural Key): 실제 데이터에서 고유값(주민번호, 이메일 등)
		- 장점: 실제 의미가 있는 데이터
		- 단점: 변경 가능성 있음
